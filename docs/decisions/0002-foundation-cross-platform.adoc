== Context

paint.type is a fresh project. Decisions taken now propagate forward indefinitely. Two things are non-negotiable for this project to mean what it is supposed to mean:

1. **It must run anywhere people are.** Linux (every distribution), macOS, Windows, Android, iOS, BSD (all flavours), MINIX, RISC-V hosts, anything that can host a typed-wasm runtime. Across architectures: x86_64, aarch64, armv7, riscv64, ppc64le, wasm32, wasm64. No decision is permitted to narrow this set.

2. **It must exploit what each host offers.** A machine with a GPU should use the GPU. A machine with an NPU/ANE/tensor cores should use those. A machine with FPGA fabric, audio DSPs, hardware crypto, vector ISAs, physics co-processors â€” all of it, where present. Absent any of them, the same code path runs (slower) on a generic CPU backend.

These two requirements together rule out platform-specific app frameworks, single-target language stacks, monolithic GPU bindings, and architectural choices that hard-code a particular OS or accelerator family.

== Decision

paint.type adopts a **unified backend/kernel pattern** modelled on https://github.com/hyperpolymath/Axiom.jl[hyperpolymath/Axiom.jl], adapted to this project's language stack.

=== The unified backend pattern (Axiom-derived)

* **One abstract operation surface.** All compositing, blending, brush, filter, transform, I/O, network, display, print and peripheral operations are named once, at the ABI layer (Idris2). Concrete implementations are registered backends.
* **Many concrete backends, one signature each.** A backend is a value implementing the abstract operations. Backends are scoped to a runtime target (e.g. `CpuBackend`, `NvidiaCudaBackend`, `AppleSiliconBackend`, `WebGpuBackend`, `FpgaPciBackend`, `DspAlsaBackend`), not to an operation class.
* **Capability flags, not directory carve-ups.** Each backend reports a structured capability set: which kernel classes (DSP, FPGA, audio, math, GPU, physics, crypto, I/O, vector, tensor) it can serve and at what precision. The dispatcher uses these flags to pick the best backend per operation per call.
* **Reference + accelerated tiers, transparent fallback.** Every operation has a reference implementation on `CpuBackend`. When an accelerator backend is selected but does not implement a specific operation, the dispatcher falls back to the reference path, emitting a single self-healing diagnostic. No call site changes.
* **Detection at start, re-detection on hot-plug.** The runtime probes available backends at startup and on capability-change events (GPU plugged in, network appearing, MIDI controller attached, â€¦). A `capability_report` structure is queryable and observable.
* **Backends are conditionally loaded.** Like Julia's package extensions, a backend's full implementation is loaded only when the host carries the underlying capability. Missing libraries are not errors â€” they reduce the capability set.

=== Language and target stack

[cols="1,3"]
|===
| Layer | Choice and reason

| Application language
| **AffineScript** (https://github.com/hyperpolymath/affinescript). Algebraic effects let the compositing/brush/network/IO traffic classes be expressed without entangling latency-critical paths with throughput-oriented ones. Affine types prevent buffer aliasing at the application layer.

| Abstract ABI / operation surface
| **Idris2.** The abstract `Backend` interface, its capability descriptor, the dispatch totality proofs, and the cross-language layout proofs all live here. Idris2 is the only place in the project where proofs are written; the rest of the stack consumes generated headers.

| FFI / kernel layer
| **Zig.** Every concrete backend is a Zig module exporting the operation set via the C ABI declared by Idris2. Zig is the only place we cross into native, vendor-specific code (CUDA, ROCm, Metal, Vulkan, WebGPU, FPGA toolchains, audio servers, â€¦).

| Unified network/API surface
| **Zig.** A single `paint_api` server, written in Zig, exposes the same operation surface over multiple wire protocols: REST, GraphQL, gRPC, Server-Sent Events, Bebop. Adding a protocol means adding an adapter; it does not duplicate the operation surface.

| Portable compilation target
| **typed-wasm** (https://github.com/hyperpolymath/typed-wasm). The AffineScript application compiles to a typed-wasm module. Backends register their own typed-wasm imports; the multi-module type checker verifies the boundary at build time, before any device touches it.

| Canonical distribution artifact
| **OCI multi-arch container.** The buildable, runnable, testable unit of paint.type from day one is a container image addressed by digest, built for every supported architecture. Native binaries are downstream products extracted from the container, not parallel build paths.
|===

=== Memory and topology awareness

The dispatcher is **topology-aware** from the start:

* CPU NUMA nodes, hyperthread groups, performance/efficiency core mix
* Available system RAM, page size, hugepages availability
* VRAM amount per GPU device, plus unified-memory hosts (Apple Silicon, Adreno)
* PCIe/NVLink/Infinity-Fabric/UMA topology between accelerator devices
* Persistent-memory tiers (CXL, Optane-style) where present
* Network reachability cost vector (loopback / LAN / WAN / metered)

Topology is exposed as a structured value alongside the capability report; AffineScript code may consult it to decide tile placement, batch sizing, and offload thresholds.

== Constraint

No design decision in this repository may:

1. Hard-code a specific operating system, libc, or window system in the application layer.
2. Make a kernel-class backend mandatory. CPU reference is always sufficient to run.
3. Limit the supported architecture set without an ADR that supersedes this one.
4. Bypass the Idris2-defined operation surface to call into a backend directly from AffineScript.

== Consequences

Positive::
* Adding a new accelerator means writing one Zig file and registering it. No app-layer changes.
* Adding a new protocol to the API surface means writing one adapter. No operation duplication.
* Adding a new OS target means producing the container for that target and verifying CI. No source-tree restructuring.
* Verification scales: Idris2 proofs cover the dispatch and ABI; per-backend correctness is a numerical-equivalence test against `CpuBackend`.

Negative / costs::
* The CPU reference backend must be implemented first and kept correct forever. It is the oracle for every other backend.
* The capability descriptor in Idris2 becomes a non-trivial schema; getting it wrong is expensive to fix later.
* `paint_api` becomes a single point with five protocol adapters. Discipline is needed to keep the adapters thin.

Neutral::
* Existing `src/ephapax/` (Rust) and `src/interface/ffi/` (Zig) become specific backends under the unified scheme rather than the primary build path. `src/ephapax/` is retained for now as a reference Rust client; the canonical FFI lives in the per-backend Zig modules.
